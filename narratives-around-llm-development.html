Exaggerated Narratives Around LLM Development

22 May 2026 · 6 min · engineering

Update: Anthropic posted a chart on June 4, 2026 indicating that its engineers, on average, "ship 8x as much code per quarter," which does not by itself imply an 8x productivity gain. This does not really refute my argument, since 8x and 100x are quite different numbers.

The common narrative right now is that great engineers are 100x more productive. I truly believe AI is an amplifier of intelligence and productivity, and therefore the gap between the most and least productive engineers has significantly increased.

That said, I want to point out that this alleged 100x amplification is based on samples that do not represent the full picture.

It is quite easy to rewrite an existing system in a new language. But is that really a paradigm shift, and should it be used as an example? It has been possible to translate documents using Google Translate for quite a few years now. The quality of those translations was often mediocre, and I would argue that the quality of code produced by many LLM-driven rewrites exhibits similar patterns.

LLMs make it quite easy to prototype. But should we treat an increase in prototyping speed as a proper measurement of software development velocity? If we measure development speed in lines of code — a very rough metric, but still a useful one — it is no secret that developers building new systems from scratch often produce more code than developers working on existing systems. They are not constrained by existing functionality or by more stringent reliability requirements.

At this point, LLMs cannot come up with novel ideas on their own. You have to figure them out and set them in writing. There is no such thing as an LLM producing breakthrough functionality in response to a prompt like “write me a system.” I do not have specific research references for this, but personal experience shared by many engineers suggests that LLMs tend to produce averaged commonality — applications that look and behave alike. Too many websites now are indistinguishable from each other. At least before, the sameness was often distorted by individual inexperience: people copied designs too directly, or made awkward attempts to make them look at least somewhat unique.

I would argue that 100x code generation is achievable only if you have provided a clear pattern and an existing set of instructions or specifications, or if you are trying to create something that was already known to the LLM at the time of training. There may be exceptions in carefully picked samples, but the multiplier would likely be much lower in the long run. For new systems, that pattern to follow does not exist, and the time needed to define specifications is quite substantial.

I also do not believe in such high multipliers for reliable systems, because the math simply does not work unless you skip or significantly narrow human participation in reviewing LLM output — or unless you blindly trust the produced output. Let’s assume that before the LLM era, you could produce one page of code. Now you end up with 100 pages. That is 100 pages you now have to read and comprehend.

Reading 100 pages of a book may not be hard, and might even be entertaining when reading fiction. But in our field, reading new code requires mental work. You are not supposed to read code passively. You are supposed to understand what it does, apply critical thinking, challenge assumptions, and verify it.

Another argument against this narrative is that the majority of software work involves modifying existing systems. Let’s assume you come to work and have to complete several existing items in your backlog. If they are simple enough, in the pre-LLM era you might have finished a few of them if you were very productive. Would you be able to finish, for example, 20 of them, even if an LLM produced the code instantly? I doubt it. Context switching has its own cost and takes time.

You may imagine having an amazing orchestrator to which you can dump all of these tasks, and it completes them without your involvement at all. But that is a scenario that does not imply the existence of software engineers at all.

A similar problem appears in the narrative around code review. Some of these claims sound reasonable on the surface, but become much less impressive once you unpack them. For example, when people say that engineers now review code created for them by LLMs, what they often mean is that the same person who requested the code is now approving it.

That is not the same review process. A review process without another independent party involved is much closer to self-review, which good engineers were already doing before asking others to review their changes.

You may argue that abandoning peer review better suits the new paradigm. After all, software existed without peer reviews for a long time. Fair. But that only means independent reviews are being eliminated for the sake of productivity gains. Is it worth it? Maybe, if all your engineers are exceptional or the cost of error is low.

Another common narrative is that managers should manage more people. It makes me wonder what kind of tools make managers much more efficient. If your communications can be handled by an LLM, was there any value in those communications? If specs are now created entirely by an LLM, what is so valuable about those specs? Has the speed of communication between people increased enough to justify this?

Software engineers now have a set of tools that allow them to speed up their work. What kind of tools do managers have that could drastically improve their productivity?

Another ingenious argument I have heard is that managers should write code. I think this is a naive idea because we still live in an environment where human labor is a factor. Asking people to do work they are not trained to do, and expecting the same level of productivity, is a mistake. No matter how intelligent a person is, all else being equal, that person would not reach the productivity level of someone experienced for quite some time — and most likely, for a very long time.

To conclude, many changes have happened recently, but too many things people proclaim to be true are not real. They are based on misunderstanding and exaggeration of what is actually happening in the industry.
