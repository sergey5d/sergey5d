Notes on idempotency keys that won't haunt you

14 May 2026 · 9 min read · distributed systems

Most retry bugs in payments don't come from missing idempotency. They come from idempotency that's almost right — the kind that survives the obvious replays and fails the interesting ones. Here are four patterns I've watched teams reinvent, and one that's actually worth keeping.

1. The "request body is the key" trap

Hashing the request body and using that as your idempotency key looks elegant for about six weeks. Then a client adds a request_id field for their own logging, the hash changes, and you charge the customer twice. The lesson: idempotency keys are client-owned, not server-derived. The server's job is to honor them, not invent them.

2. Storing only the key, not the response

The first 80% of an idempotency layer is "have we seen this key before?". The next 19% is "what did we tell the client last time?". If you only store the key, the second request gets a different response shape than the first — same outcome, different envelope. That's a bug that surfaces a week later, in a client retry loop, at 2am.

CREATE TABLE idempotency (
  key            text PRIMARY KEY,
  request_hash   bytea NOT NULL,
  response_body  jsonb NOT NULL,
  response_code  smallint NOT NULL,
  created_at     timestamptz NOT NULL DEFAULT now(),
  expires_at     timestamptz NOT NULL
);

3. Mismatched bodies under the same key

What happens when a client sends two different requests under the same idempotency key? The wrong answer is "return the first response anyway". The right answer is a 409 Conflict with a clear message. Stripe's done this for years and it's still the cleanest contract I've seen.

If two requests claim to be the same retry but aren't, one of you is lying. Refuse to guess which one.

4. Expiry that's too short, then too long

I've seen 24-hour windows ("retries shouldn't take longer than that, right?") and I've seen 90-day windows ("storage is cheap"). Both are wrong. The right window matches your longest legitimate retry path — usually a queue's max age plus a safety margin. For us, that's 72 hours. Your number will differ; don't copy mine.

The one pattern worth keeping

Make the idempotency layer a separate concern from the business logic. It's a thin table, a hash check, and a response cache — that's it. Resist the urge to make it "smart". Smart idempotency layers are how you end up with a 4-month migration to fix a behavior nobody documented.
